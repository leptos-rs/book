# Quick reference of asynchronous closures/futures

With all the different types of asynchronous closures and futures being introduced, a summary of what they do at where and when is useful to keep in mind.  The following table briefly describes when the code is executed for each of the asynchronous closure types that have been introduced.  Note that each type behaves differently on the server and on the client.

| Type | Server-side rendering | During Hydration to prepare for CSR | Client-side rendering |
|------|-----------------------|-------------------------------------|-----------------------|
| Server Functions (Server) | Invoked as a normal function directly, with the code defined in the body executed with the results returned to the caller. | - | (Invoked by responding to the HTTP request made to the server function's endpoint, where the response will be the serialized result.) |
| Server Functions (Client) | - | Not Executed (when defined inside resource async closures). | Makes an HTTP request to the server function's endpoint, the response body will be deserialized into the result. Do note that the code in the function body is never compiled into the client. |
| Resources (Server) | Their async closures are executed in *parallel* when possible by the async executor; results are encoded into the response body inside `<script>` tags. | - | - |
| Resources (Client) | - | Not executed; the `<script>` tags within the response body provides the results. | Their async closures are executed whenever required by the reactive system to return any new results. |
| Local Resources (Client Only) | - | Not executed. (Will be executed once after hydration to provide the initial value.) | Their async closures are executed whenever required by the reactive system to return any new results. |
| Suspend (Server) | These futures are executed in *parallel* when possible by the async executor; how and when they stream their output over the connection may be modified by the selected `SsrMode`. | - | - |
| Suspend (Client) | - | These futures are fully executed in the client as part of hydration; returned `view! {}` will be used to hydrate the HTML. | These futures are executed whenever required by the reactive system; returned `view! {}`  will be rendered wherever they are defined. |

A useful note to keep in mind is that there are up to three different stages in the rendering pipeline, executed on both the server and the client.  One consequence of this are the potential for hydration bugs or unexpected program behavior.  For example, if a resource sets a signal, it might behave perfectly fine under SSR and CSR, but will not behave in the expected manner during hydration, as its code is _not_ executed during that rendering stage.  Another example is that server functions may have access to a context during SSR (as it's invoked as a typical async function within the rendering/reactive context) and might not have access to it when acting as a standalone endpoint (as it's just an ordinary endpoint without any rendering context).  The above is really just a quick reference of the behavior of the different types under the typical documented configuration, as combining or nesting the above in some other manner can cause other unexpected behaviors.
